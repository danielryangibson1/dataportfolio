BTC Price Notifications

Finding the optimal threshold for notifying users about movements in price

Push notifications are one of the primary levers for driving user engagement in many businesses, but send too many, and users disable them entirely. The question is: at what price movement threshold should a notification be sent in order to sustain engagement without tipping users toward turning them off?

The answer?

A threshold of approximately 2.8–3.7% aligns with published research on roughly every 2–3 days being the optimal notification frequency for fintech apps. The final choice within that range is a product decision: whether to lean toward higher engagement frequency (2.8%) or lower opt-out risk (3.7%).

Methodology

One year of hourly BTC price data was sourced via the Binance API. For each hour, the maximum price swing within the preceding 24-hour window was calculated (the high divided by the low, minus one, in order to get the percentage change). This is deliberately more sensitive than a simple point-to-point comparison (now vs 24 hours ago): by capturing the widest movement at any point in the last 24 hours rather than between two fixed points in time, it reflects the kind of move a user would actually have noticed and been interested in.

A notification will send when a price swing exceeds a given threshold, subject to a 24-hour cooldown that prevents repeated triggers during sustained volatility. The cooldown is a key design decision — it ensures the system drives engagement at a consistent cadence rather than clustering alerts around a single market event.

The Thresholds

Thresholds were derived programmatically from the actual price swing range observed in the dataset, with outliers trimmed from both ends. The five shown in the dashboard — 2.8% through 6.5% — are a presentational subset chosen to illustrate the spread. In practice, a more precise recommendation could be produced by narrowing the range and increasing the number of threshold steps, allowing the analysis to hone in tightly around the optimal value rather than mapping the broader curve.

Grounding in Research

Industry data on push notification frequency consistently points to a sweet spot of roughly every 2–3 days for apps in the financial space. Research shows that even one push notification per week leads to around 10% of users disabling notifications.

At a 2.8% threshold, the simulation produces a notification approximately every 2 days.

At 3.7%, it becomes every 3.1 days.

The modelled output maps closely to the research-backed range.

The Final Call

Both thresholds are defensible, and at this point the data has done its job. The final call comes down to the preference of the final decision maker.

For the fellow data nerds

The full pipeline runs inside Power Query. A Python script makes a paginated call to the Binance API, fetching one year of hourly BTCUSDT candles in batches of 1,000. The output of that script is immediately passed as the input to a second Python script, which calculates the rolling price swings and simulates notification triggers across all thresholds. Power Query then splits the result into three tables: the raw price data, the notification trigger flags, and a lookup table mapping column positions to threshold values.

Threshold Trigger Calculation — Python
import pandas as pd
import numpy as np

df = dataset
df['DateTime'] = pd.to_datetime(df['DateTime'])
df = df.sort_values('DateTime').reset_index(drop=True)
df = df.set_index('DateTime')

rolling_max = df['BTC Price'].rolling('24h').max()
rolling_min = df['BTC Price'].rolling('24h').min()
df['price_swing'] = (rolling_max / rolling_min - 1)
df = df.reset_index()

min_threshold = df['price_swing'].min()
max_threshold = df['price_swing'].max()
thresholds = np.round(abs(np.linspace(min_threshold, max_threshold, 18)), 3)[3:-7]

for threshold in thresholds:
    notifications = []
    last_notification_time = None

    for _, row in df.iterrows():
        if abs(float(row['price_swing'])) >= threshold:
            if last_notification_time is None or \
               (row['DateTime'] - last_notification_time).total_seconds() >= 86400:
                notifications.append('Yes')
                last_notification_time = row['DateTime']
            else:
                notifications.append('No')
        else:
            notifications.append('No')

    df[f'{threshold}%'] = notifications

df = df.drop(columns=['price_swing'])

lookup = pd.DataFrame({
    'col_name': [f'col_{i+1}' for i in range(len(thresholds))],
    'threshold_value': thresholds
})

The Python script names threshold columns using their actual percentage values (e.g. 2.8%). Those names are dynamic — change the thresholds and the column names change with them. To ensure the Power BI model never breaks on refresh, the columns are immediately renamed by position in Power Query: col_1, col_2, and so on. The Lookup table carries the mapping between position and actual threshold value, and drives a disconnected slicer in the report. DAX then reads the slicer selection via SELECTEDVALUE and uses SWITCH to filter against the correct positional column. The result: update the thresholds in Python, refresh, and everything — the slicer labels, the chart markers, the average days calculation — updates automatically without touching a single measure.

BTC Price Notification Triggered =

VAR currently_selected_threshold =
    SELECTEDVALUE ( slic_BinancePriceData_BTC[col_name] )

VAR count_of_yes =
    CALCULATE (
        COUNTROWS ( fact_BinancePriceNotifications_BTC ),
        SWITCH (
            currently_selected_threshold,
            "col_1", fact_BinancePriceNotifications_BTC[col_1] = "Yes",
            "col_2", fact_BinancePriceNotifications_BTC[col_2] = "Yes",
            "col_3", fact_BinancePriceNotifications_BTC[col_3] = "Yes",
            "col_4", fact_BinancePriceNotifications_BTC[col_4] = "Yes",
            "col_5", fact_BinancePriceNotifications_BTC[col_5] = "Yes"
        )
    )

VAR btc_Price = [AVG {BTC Price}]

RETURN
    IF ( count_of_yes = 0, BLANK(), btc_Price )
Avg Days Between Notifications — DAX
Avg Days Between Notifications =

VAR currently_selected_threshold =
    SELECTEDVALUE ( disc_BinancePriceData_BTC[col_name] )

VAR dates_with_yes =
    CALCULATETABLE (
        DISTINCT ( fact_BinancePriceNotifications_BTC[Date] ),
        SWITCH (
            currently_selected_threshold,
            "col_1", fact_BinancePriceNotifications_BTC[col_1] = "Yes",
            "col_2", fact_BinancePriceNotifications_BTC[col_2] = "Yes",
            "col_3", fact_BinancePriceNotifications_BTC[col_3] = "Yes",
            "col_4", fact_BinancePriceNotifications_BTC[col_4] = "Yes",
            "col_5", fact_BinancePriceNotifications_BTC[col_5] = "Yes"
        )
    )

VAR days_since_last_yes =
    ADDCOLUMNS (
        ADDCOLUMNS (
            dates_with_yes,
            "@prev_date", WINDOW ( -1, REL, -1, REL, ORDERBY ( fact_BinancePriceNotifications_BTC[Date], ASC ) )
        ),
        "@days_between",
            IF (
                NOT ISBLANK ( [@prev_date] ),
                [Date] - [@prev_date]
            )
    )

RETURN
    AVERAGEX ( days_since_last_yes, [@days_between] )
